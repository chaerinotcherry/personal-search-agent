# 데이터베이스 기초와 실무

## 관계형 데이터베이스 (RDBMS)

관계형 데이터베이스는 테이블, 행, 열로 데이터를 구조화하며 SQL로 조작합니다. PostgreSQL, MySQL, SQLite가 대표적입니다. ACID(원자성, 일관성, 격리성, 지속성) 속성을 보장해 트랜잭션 처리에 신뢰성이 높습니다. 외래키(Foreign Key)로 테이블 간 관계를 정의하고 참조 무결성을 유지합니다.

## 정규화

데이터 중복을 줄이고 이상(Anomaly)을 방지하기 위해 정규형(Normal Form)을 적용합니다. 제1정규형(1NF): 각 컬럼은 원자값만 가져야 합니다. 제2정규형(2NF): 부분 함수 종속을 제거합니다. 제3정규형(3NF): 이행적 함수 종속을 제거합니다. 지나친 정규화는 JOIN이 많아져 성능이 저하될 수 있어 실무에서는 적절한 비정규화도 고려합니다.

## 인덱스 설계

인덱스는 검색 속도를 높이지만 쓰기 성능과 저장 공간을 희생합니다. B-Tree 인덱스가 가장 일반적이며 범위 검색에 효과적입니다. Hash 인덱스는 등가 비교에 빠릅니다. 복합 인덱스(Composite Index)는 자주 함께 사용되는 컬럼에 적용하며, 컬럼 순서가 성능에 중요합니다. `EXPLAIN ANALYZE`로 쿼리 실행 계획을 확인해 인덱스 효과를 검증합니다.

## 트랜잭션과 격리 수준

트랜잭션 격리 수준은 동시 접근 시 데이터 일관성과 성능의 트레이드오프를 결정합니다. READ UNCOMMITTED는 더티 리드를 허용하고, READ COMMITTED는 더티 리드를 방지합니다. REPEATABLE READ는 같은 트랜잭션 내 반복 조회에서 동일한 결과를 보장합니다. SERIALIZABLE은 가장 엄격하지만 성능 오버헤드가 큽니다. 실무에서는 READ COMMITTED 또는 REPEATABLE READ를 주로 사용합니다.

## ORM vs Raw SQL

ORM(SQLAlchemy, Django ORM)은 개발 생산성을 높이고 SQL 인젝션을 방지하지만, 복잡한 쿼리에서 비효율적인 SQL이 생성될 수 있습니다. 특히 N+1 쿼리 문제에 주의해야 합니다. 성능이 중요한 쿼리는 `text()` 또는 Raw SQL로 직접 작성하고, 일반 CRUD는 ORM을 활용하는 혼합 전략이 실용적입니다.

## 커넥션 풀링

데이터베이스 연결 생성은 비용이 크므로 커넥션 풀을 사용해 연결을 재사용합니다. SQLAlchemy의 `create_engine(pool_size=10, max_overflow=20)`처럼 풀 크기를 설정합니다. 풀이 가득 찼을 때 대기 타임아웃을 설정해 데드락을 방지합니다. 비동기 환경에서는 `asyncpg` + `SQLAlchemy AsyncEngine`을 사용합니다.
