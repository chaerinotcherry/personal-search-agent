# 백엔드 아키텍처 설계 원칙

## 레이어드 아키텍처

백엔드 시스템은 일반적으로 프레젠테이션 레이어(API), 비즈니스 로직 레이어(서비스), 데이터 레이어(리포지토리)로 나뉩니다. 각 레이어는 독립적이어야 하며, 상위 레이어만 하위 레이어에 의존해야 합니다. 이 원칙을 지키면 테스트가 쉬워지고 변경의 영향 범위가 줄어듭니다.

## 도메인 주도 설계 (DDD)

도메인 주도 설계는 비즈니스 도메인을 중심으로 소프트웨어를 설계하는 방법론입니다. 핵심 개념으로 엔티티(Entity), 값 객체(Value Object), 집계(Aggregate), 리포지토리(Repository), 도메인 서비스가 있습니다. 복잡한 비즈니스 로직을 가진 시스템에서 DDD를 적용하면 코드가 비즈니스 언어와 일치해 유지보수가 용이해집니다.

## SOLID 원칙

단일 책임 원칙(SRP): 클래스는 하나의 변경 이유만 가져야 합니다. 개방-폐쇄 원칙(OCP): 확장에는 열려있고 수정에는 닫혀있어야 합니다. 리스코프 치환 원칙(LSP): 자식 클래스는 부모 클래스를 대체할 수 있어야 합니다. 인터페이스 분리 원칙(ISP): 클라이언트가 사용하지 않는 인터페이스에 의존하지 않아야 합니다. 의존성 역전 원칙(DIP): 고수준 모듈이 저수준 모듈에 의존하지 않고, 둘 다 추상화에 의존해야 합니다.

## 이벤트 기반 아키텍처

서비스 간 결합도를 낮추기 위해 이벤트 기반 통신을 사용합니다. 주문 완료 이벤트가 발행되면 알림 서비스, 재고 서비스, 분석 서비스가 독립적으로 이벤트를 구독해 처리합니다. Kafka, RabbitMQ, Redis Pub/Sub 등이 메시지 브로커로 자주 사용됩니다. 이벤트 소싱 패턴과 결합하면 시스템 상태 변화를 완전히 추적할 수 있습니다.

## CQRS 패턴

Command Query Responsibility Segregation은 쓰기(Command)와 읽기(Query)를 분리하는 패턴입니다. 쓰기는 정규화된 데이터베이스에, 읽기는 최적화된 뷰(View DB, 캐시)에서 처리합니다. 트래픽이 많은 서비스에서 읽기 성능을 극대화할 수 있으나, 데이터 동기화 복잡도가 증가합니다.

## 마이크로서비스 vs 모놀리식

모놀리식은 배포가 간단하고 서비스 간 통신 오버헤드가 없지만, 팀이 커지면 코드베이스 관리가 어려워집니다. 마이크로서비스는 서비스별 독립 배포·확장이 가능하지만 분산 시스템의 복잡성(네트워크 실패, 분산 트랜잭션)을 처리해야 합니다. 보통 모놀리식으로 시작해 병목이 생기는 서비스부터 분리하는 전략을 권장합니다.
